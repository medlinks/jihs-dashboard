### ① 合并周 PDF 结构检查（35/35 通过）

对每个合并 PDF 验证三项：
- 目录页（含 "第XX週のデータ" 导航文字）**不会**被误当作数据页处理；
- w1 和 w2 的数据页都能被识别（两个 `第XX週のデータ` 标记都找到）；
- w1/w2 分配的数据页数对称。

## 4. 未来健壮性设计建议

现有 `extract_jihs_fixed.py` 的核心策略是"内容驱动"而非"页码驱动"——这已经是正确方向。下面列出当前已有的防御点与建议加强项。

### 4.1 现有防御（已实现）

| 策略 | 实现位置 | 作用 |
|------|---------|------|
| 列头驱动表格定位 | `extract_page`：寻找 `報告数 + 累積/定点当り` 行 | 页面顺序变动不影响 |
| Voronoi 列分配 | `voronoi_assign` | 字符渲染偏移不影响列归属 |
| 周标记驱动合并 PDF 分段 | `extract_year`：`第XX週のデータ` 字符串触发 current_wk 切换 | 无需依赖页码 |
| skip_page 黑名单 | `should_skip_page` | 兽医、性别细分、MERS 合并表格自动跳过 |
| 累积列过滤 | `ri_list` 只取 `報告数` 列 | 累积数不会误当作当周报告 |
| 文件名匹配 | `idwr{year}-(\d+)(?:-(\d+))?\.pdf` | 自动检测合并 PDF |

### 4.2 建议加强（针对未来可能的格式变动）

**A. 更鲁棒的周标记识别**

```python
# 现在：
if f'{w2}週のデータ' in page_text: current_wk = wk2
# 建议：正则 + 多种写法兼容
import re
MARK_PAT = re.compile(r'第\s*(\d+)\s*週のデータ|第\s*(\d+)\s*週\s*データ')
for m in MARK_PAT.finditer(page_text):
    wk = int(m.group(1) or m.group(2))
    if wk == w1: current_wk = wk1
    elif wk == w2: current_wk = wk2
```

**B. 3+ 周合并场景的前瞻支持**

目前文件名 regex 只识别 2 周合并。建议：

```python
# 支持 idwr2025-01-02-03.pdf 这种（虽然 JIHS 历史上未出现）
m = re.search(r'idwr(\d{4})-((?:\d+-?)+)\.pdf', filename)
weeks = [int(w) for w in m.group(2).split('-')]
```

**C. 页数变动防护**

JIHS 在 2018、2023 各增加了 1-2 页。建议引入 **"页数校验 + 警告而非失败"**：

```python
EXPECTED_PAGES_PER_WEEK = {2013: 13, 2014: 12, 2015: 12, 2018: 13, 2023: 14, 2024: 14}
if abs(actual_pages - expected) > 2:
    print(f"WARNING: {pdf} pages={actual_pages} (expected ~{expected})")
    # 不阻塞，但记录到日志
```

**D. 内容完整性回归测试**

建立 `tests/reference_values.yaml` 保存若干高置信度的 (年,周,疾病,都道府县) = 值 三元组。每次重新提取后用 CI 模式运行：

```yaml
- year: 2025
  week: 17
  disease: 梅毒
  prefecture: 東京都
  value: 36   # 手动核对过
```

一旦格式变动导致某值偏移，立即暴露。

**E. `overrides.json` 手动修正接口**

极端情况（PDF 扫描错字、官方修订）下允许人工覆盖：

```json
{
  "2024": {
    "第35週": {"梅毒": {"沖縄県": 3}}
  }
}
```

合并到 `full_dashboard_data.json` 之前应用 overrides。

**F. PDF 下载 + 提取 + 验证一体化流水线**

建议加一个 `scripts/pipeline.py`：
1. 扫 JIHS 网站有新 PDF → 下载到对应年份文件夹
2. 运行 `extract_jihs_fixed.py`
3. 对比新 Excel 与旧 Excel 的相邻周变化（同比、环比）
4. 如果变化 > 阈值或出现 NaN，发告警

**G. 列头白名单化 + fail-fast**

如果未来 JIHS 把 `報告数` 改成 `報告件数` 或 `新規報告`，当前代码会静默返回空字典（导致该页数据丢失、不报错）。建议：

```python
KNOWN_HEADERS = {'報告数', '累積', '定点当り', '定点当たり'}
UNKNOWN_HEADER_WARNING_THRESHOLD = 3  # 连续 3 页找不到已知列头就报警
```

### 4.3 优先级建议

1. **立刻做**：D（回归测试 yaml）、G（列头警告）
2. **近期做**：A（正则化标记）、C（页数警告）
3. **需要时做**：B、E、F
